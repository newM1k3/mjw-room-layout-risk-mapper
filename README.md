# MJW Room Layout Risk Mapper

A structured, step-by-step tool for escape room designers to map physical room layouts, identify design risks, and export actionable reports. It guides designers through project setup, zone definition, element placement, and risk analysis — producing a prioritised risk map from the canvas. Optional **PocketBase cloud saves** preserve projects across sessions and devices.

## What It Does

Unlike generic floor-plan or diagramming tools, this app uses escape-room-specific concepts — zones, elements, flow paths, and risk categories — to surface layout problems before build day.

| Concept | Meaning |
| :---- | :---- |
| **Project** | A named room design with metadata such as player count, room dimensions, and difficulty target. |
| **Zone** | A defined spatial area within the room — e.g. "Entrance Corridor," "Puzzle Alcove," "Finale Area." |
| **Element** | A prop, lock, clue station, or interaction point placed inside a zone. |
| **Risk Issue** | A flagged layout concern generated by the risk engine — e.g. crowding, dead-end flow, or inaccessible element. |
| **Risk Map** | The full annotated canvas showing zones, elements, and overlaid risk indicators. |
| **Summary Panel** | A collated view of all risk issues with severity ratings and recommended fixes. |

**Key interactions:**

- Complete the guided setup flow step-by-step from welcome through to export.
- Define zones and place elements on the layout canvas.
- Run the risk engine to generate annotated risk issues across the canvas.
- Review individual risk issue cards with severity, affected zone, and recommended action.
- Track overall progress through the step rail at the top of the app.
- Export the completed risk map and summary report for sharing with build teams.
- Optionally save and reload projects via PocketBase cloud persistence.

## How to Use

The app opens with a welcome screen and walks designers through a linear progression: project metadata, zone construction, element placement, risk mapping, and export. Each step is accessible from the top progress rail. Complete the project setup form first to unlock the zone builder, then add elements before running the risk engine. The summary panel aggregates all flagged issues in one place. Use the export panel to download the risk map for handoff to build or set-dressing teams.

The tool is designed primarily for desktop use where canvas interactions, zone drawing, and detailed form editing work best. The layout is functional on tablet-sized screens for review purposes.

| Variable | Required? | Scope | Enables | Description |
| :---- | :---- | :---- | :---- | :---- |
| `VITE_POCKETBASE_URL` | Optional | Frontend/public | PocketBase sign-in and cloud project saves | Public PocketBase/PocketHost URL used for user authentication and project record CRUD. Example: `https://immersive-kit.pockethost.io`. |

## Saved Projects and PocketBase Cloud Saves

The app works fully with **no environment variables**. In local-only mode, project state is held in memory for the current session, and designers can still complete the full step flow, generate risk maps, and export reports.

Cloud saves are optional. When `VITE_POCKETBASE_URL` is configured, authenticated users can persist project records — including zone definitions, element placements, and risk results — to a PocketBase collection. Normal user authentication runs through the public PocketBase URL; **no PocketBase superuser token is placed in frontend code**.

### Recommended `room_projects` Collection

Create a PocketBase collection named `room_projects`. The current implementation expects authenticated users to own their own records. For the MJW canonical schema, configure the following fields.

| Field | Type | Notes |
| :---- | :---- | :---- |
| `title` | text | Display name shown in the project list. |
| `description` | text | Optional room design notes. |
| `owner` | relation to `users` | Should point to the authenticated user. |
| `project_json` | json | Stores zones, elements, canvas config, and risk results. |
| `room_dimensions` | text | Width × depth notation for quick display. |
| `player_count` | number | Target player capacity for risk scoring. |
| `difficulty` | select | Recommended values: `beginner`, `standard`, `expert`. |
| `visibility` | select | Recommended values: `private`, `shared`. |
| `version` | number | Incremented on saves for conflict detection. |
| `created` | system field | Managed by PocketBase. |
| `updated` | system field | Managed by PocketBase. |

Recommended collection rules should allow authenticated users to create records for themselves and only read, update, or delete their own records. A practical rule pattern is `@request.auth.id != "" && owner = @request.auth.id` for user-scoped list/view/update/delete rules. The create rule should require authentication and an owner value matching the authenticated user.

## Netlify Deployment

The `netlify.toml` at the project root configures the Vite build and static routing. To deploy on Netlify, connect this GitHub repository and use the following production settings.

| Setting | Value |
| :---- | :---- |
| Build command | `npm run build` |
| Publish directory | `dist` |
| Node/package install | Netlify default Node environment with `npm install` |

```toml
[build]
  command = "npm run build"
  publish = "dist"

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
```

Deploy first with no environment variables to confirm the local-only app works, then add `VITE_POCKETBASE_URL` for cloud project saves.

## Accessibility and Production Readiness

The step-flow UI uses a visible progress rail so designers always know where they are in the process. Risk issue cards include severity labelling and actionable recommendations rather than raw error codes. Empty and unconfigured states are explicit — the app does not fail silently when PocketBase is unavailable. The export panel produces outputs suitable for direct handoff to build and set-dressing teams.
